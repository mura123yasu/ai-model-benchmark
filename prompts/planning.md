# 検討・計画プロンプト集

---

## 1. 要件定義

### プロンプト

```
中小企業向け勤怠管理システムの要件定義書ドラフトを作成してください。

【背景】
- 対象：従業員数10〜100名規模の中小企業
- 現状：紙・Excelによる勤怠管理からの脱却を目指している
- 目的：打刻・集計・申請承認のデジタル化

【出力形式】
以下のすべての項目を含めて出力してください。

1. システム概要
   - 目的・スコープ・対象ユーザー

2. 機能要件（MoSCoW法で分類）
   - Must（必須機能）
   - Should（あると良い機能）
   - Could（余裕があれば対応する機能）

3. 非機能要件
   - 性能・可用性・セキュリティ・保守性

4. 制約条件
   - 技術的・法的・予算・スケジュールの制約

5. ヒアリング項目
   - 要件定義を進めるためにクライアントに確認すべき質問（10項目以上）
```

### 評価ポイント

- **網羅性**：5つの出力項目がすべて適切に記載されているか
- **非機能要件の現実性**：性能・可用性の数値目標（レスポンスタイム・稼働率など）が中小企業向けとして現実的か
- **ヒアリング項目の質**：表面的な質問ではなく、実際の要件定義に必要な情報を引き出せる質問になっているか

## 2. 設計支援

### プロンプト

```
タスク管理Webアプリのシステム設計を行ってください。

【要件】
- 同時接続数：最大1,000人
- 月間アクティブユーザー：10万人
- クラウド：AWS
- フロントエンド：React
- その他の技術スタックは設計の観点から最適なものを選択してください

【出力形式】
以下のすべての項目を含めて出力してください。

1. アーキテクチャ概要
   - 構成図（テキストベースの図でよい）

2. 技術スタック
   - フロントエンド・バックエンド・DB・インフラ・CI/CDの各層を明記

3. DB設計
   - 主要テーブルのER図（テキストベース）と各カラムの型・制約

4. APIエンドポイント設計
   - 主要エンドポイントを10本以上、メソッド・パス・リクエスト/レスポンス形式を含めて定義

5. スケーリング戦略
   - 負荷増加時の対応方針（水平スケーリング・キャッシュ・CDNなど）

6. トレードオフの説明
   - 採用した技術・設計の選択理由と、それに伴うデメリット・リスク
```

### 評価ポイント

- **規模感に適したアーキテクチャ**：同時接続1,000人・月間10万人という規模に対して、過不足のない構成が選択されているか（過剰設計・過少設計でないか）
- **トレードオフの言語化**：技術選択の理由とデメリットが具体的に説明されているか
- **ER図の正規化レベル**：DB設計が適切な正規化（第3正規形程度）になっているか、冗長なデータ構造や非正規化の意図が明示されているか
